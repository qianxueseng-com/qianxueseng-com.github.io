The Zen of Python, by Tim Peters

  • Beautiful is better than ugly.
  • Explicit is better than implicit.
  • Simple is better than complex.
  • Complex is better than complicated.
  • Flat is better than nested.
  • Sparse is better than dense.
  • Readability counts.
  • Special cases aren’t special enough to break the rules.
  • Although practicality beats purity.
  • Errors should never pass silently.
  • Unless explicitly silenced.
  • In the face of ambiguity, refuse the temptation to guess.
  • There should be one– and preferably only one –obvious way to do it.
  • Although that way may not be obvious at first unless you’re Dutch.
  • Now is better than never.
  • Although never is often better than right now.

网上的中文翻译

Python之禅

赖勇浩翻译

  • 优美胜于丑陋（Python 以编写优美的代码为目标）
  • 明了胜于晦涩（优美的代码应当是明了的，命名规范，风格相似）
  • 简洁胜于复杂（优美的代码应当是简洁的，不要有复杂的内部实现）
  • 复杂胜于凌乱（如果复杂不可避免，那代码间也不能有难懂的关系，要保持接口简洁）
  • 扁平胜于嵌套（优美的代码应当是扁平的，不能有太多的嵌套）
  • 间隔胜于紧凑（优美的代码有适当的间隔，不要奢望一行代码解决问题）
  • 可读性很重要（优美的代码是可读的）
  • 即便假借特例的实用性之名，也不可违背这些规则（这些规则至高无上）
  • 不要包容所有错误，除非你确定需要这样做（精准地捕获异常，不写 except:pass 风格的代码）
  • 当存在多种可能，不要尝试去猜测
  • 而是尽量找一种，最好是唯一一种明显的解决方案（如果不确定，就用穷举法）
  • 虽然这并不容易，因为你不是 Python 之父（这里的 Dutch 是指 Guido ）
  • 做也许好过不做，但不假思索就动手还不如不做（动手之前要细思量）
  • 如果你无法向人描述你的方案，那肯定不是一个好方案；反之亦然（方案测评标准）
  • 命名空间是一种绝妙的理念，我们应当多加利用（倡导与号召）

github-pages-tips

github pages 的坑

有四种page

  • User Pages site，这个应该取名叫username.github.io
  • Organization Pages site，这个取名叫orgname.github.io
  • Project Pages site owned by a user account，这个如果取名叫proa的话，就会在username.github.io/proa
  • Project Pages site owned by an organization，这个如果取名叫prob的话，就会在orgname.github.io/prob

有两种branch

  • User Pages site和Organization Pages site应该用master branch
  • Project Pages site owned by a user account和Project Pages site owned by an organization应该用gh-pages branch

绝对坑爹…

the wrong way to learn front end

学前端不是纸上谈兵

看到各种各样的前端入门指南，只能说，不要纸上谈兵了。

误区一，只看书不练习

想一想，你是愿意和一个能背出军事名著的人打仗还是和一个身经百战的人打仗。同样的，你是和一个读过很多书的人做项目还是和一个真的写了很多代码的人做项目呢？

误区二，花大量时间学习冷门的css或者html

然后并没有什么太多用处。茴字有几种写法？我不太管你知道多少种居中的办法，只要你知道一两种，并且知道他们为什么工作，就行了。

误区三，贪全

我又会指挥水军，又会指挥骑兵，又会攻城。。。估计您是都不太行吧。。。。不用想着上来就是什么都会，一项项来。写几个微信的html5页面，自然会移动端开发了。写几个支持IE的应用，自然会兼容了。

Isomorphic React render

服务器和客户端重用React代码

大家都崇拜在服务器和客户端重用React代码.所有看上去美好的事情都是有坑，上帝不是教导过你，没有免费的午餐吗？
目前碰到的坑有：

性能问题

因为React比一般的模板框架例如(mustache)要考虑更多的事情，所以在纯粹生成HTML上并没有那么有效率。有文章指出，说和mustache比起来，React要慢一个数量级。Mustache几十毫秒搞定的，React需要几百毫秒。

架构问题

很多常用的Flux库都不能够支持服务器段的渲染，因为singleton在React/Flux架构中无处不在，无时不有。这个在浏览器里面是没有任何坏处的，可是在服务器段就不行了。具体上说，store/dispatcher都最好不是singleton。

代码复杂程度

要同时支持服务器和客户端渲染是1+1>2。特别是在涉及到需要从其他地方取数据的时候，支持两个环境渲染意味着在两个环境里面都需要去拿数据。

归根结底

在两个环境里重用确实好，但是绝对要付出代价。

react native FAQ

FAQ:

Q: What is React?

A: A declarative, efficient, and flexible JavaScript library for building user interfaces. It transforms data into UI.

Q: What is React Native?

A: When transforming data into UI, the UI platform becomes an implementation detail. So React can render into DOM (web), Canvas, iOS views and Android views.

Q: Are we using web views?

A: No.

Q: Is this related to RxJava?

A: Not really.

Q: Why do I want to write native apps in JavaScript?

A: React has proven to be very effective in writing UI and Facebook wants to bring the same paradigm to native platforms. JavaScript again becomes an implementation detail. As a matter of fact, there is an Objective C++ implementation of React paradigm.

Q: Are you sure React Native is a good idea?

A: Nobody knows. That is why we want to give it a try and I heard the main Facebook app has some views written in React Native as well.

Q: What do I need to know to learn React Native?

A: It is better if you know JavaScript (React), and CSS (Flexbox). You also need to follow the guide to set up Android development environment on your machine.

zero to one

受到startup podcast的启发，我也想尽量详细的记录下来前学僧的一步一步。

目前的设计是：

  • 域名注册在google domain
  • DNS在cloudflare
  • 代码在github
  • 静态页面也在github上，不过是用github pages
  • 编辑器是用Atom

为什么用静态页面生成器?

因为前学僧主要是以内容为主，并没有太多动态的页面。

为什么用github pages?

因为免费，并且方便。
